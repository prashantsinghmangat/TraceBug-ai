# ADR 003 — rrweb DOM replay instead of video-by-default

**Status:** Accepted (v1.7)

## Context

Session replay can be pixel video (`getDisplayMedia` → `.webm`, tens of MB,
a screen-share permission prompt, opaque pixels) or DOM reconstruction.
rrweb is the de-facto open-source standard for DOM recording: snapshot +
incremental mutations, small, inspectable, maskable.

## Decision

rrweb is the default replay surface; video is opt-in. The exported viewer
inlines rrweb's Replayer runtime (regenerated by `npm run gen:rrweb`) and
reconstructs the page in a sandboxed iframe.

## Consequences

- KB instead of MB, no permission prompt, click-to-seek from console/network
  rows, and text stays selectable/inspec­table in the replay.
- Privacy composes: `maskAllInputs`, `.tb-block`/`.tb-mask`, and the
  element-level blur's `tb-mask` all act on the recorded DOM — video can
  only ever blur pixels.
- Accepted limitation: canvas/WebGL-heavy apps lose visual fidelity
  (`recordCanvas` is off for size); we say so publicly rather than hide it.
- TraceBug's own UI must be excluded via `blockSelector` — learned the hard
  way when the widget recorded itself into replays (fixed in 1.7).
